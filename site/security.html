Security & Trust

Identity assurance for autonomous agents.

Agent Identity is designed around verifiable identifiers, signed credentials, key rotation, revocation, capability scoping, reputation, and auditability.

Principle

Cryptographic identity

Each agent is represented by a stable DID and one or more verification keys. Services should verify signatures against the active DID document before trusting claims.

Agent DID
  └── DID Document
        ├── verificationMethod
        ├── authentication
        ├── assertionMethod
        └── capabilityInvocation
Principle

Least privilege capabilities

Capabilities should be scoped by resource, action, validity window, and contextual constraints such as tenant, environment, region, or maximum usage rate.

{
  "capability": "document.read",
  "resource": "papers",
  "action": "read",
  "constraints": {
    "tenant_id": "tenant_123",
    "max_requests_per_hour": 100
  }
}
Verification Flow

Before trusting an agent

1. Resolve DID
2. Validate key status
3. Verify signature
4. Check credential expiry
5. Check revocation registry
6. Evaluate capabilities
7. Review trust score
8. Write audit event
Revocation

Compromise response

Credentials, keys, grants, and agents should be revocable. Downstream systems should treat revocation checks as mandatory for sensitive operations.

POST /v1/revocations
{
  "target_type": "credential",
  "target_id": "urn:uuid:...",
  "reason": "Credential compromised"
}
Auditability

Every identity event should be traceable

Agent creation, key rotation, credential issuance, trust recalculation, capability grants, and revocations should all produce audit events.

agent.created
key.rotated
credential.issued
capability.granted
credential.revoked
trust.recalculated

Trust should be verified, scoped, revocable, and observable.

Agent Identity provides the foundation for secure agent-to-tool and agent-to-agent interactions.
